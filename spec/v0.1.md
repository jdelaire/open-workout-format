# Workout Day Markdown v0.1

Status: Draft

Workout Day Markdown is a Markdown-based format for describing one complete workout day. It is intended as an open interchange format between coaches, athletes, apps, and AI systems that convert messy workout input into structured training instructions.

## Scope

A v0.1 document describes what should be completed for a single workout day.

It may include:

- warm-ups and cooldowns;
- strength and hypertrophy prescriptions;
- supersets, tri-sets, circuits, and complexes;
- AMRAP, EMOM, interval, tabata, and for-time work;
- tests and max-effort blocks;
- coaching notes, tempo, rest, RPE, RIR, load, distance, duration, and score instructions.

It does not describe:

- a full multi-week calendar;
- athlete state or readiness;
- equipment inventory;
- substitutions chosen at runtime;
- logged completion data.

## Design Principles

1. **Markdown first:** the document must remain useful in a normal Markdown viewer.
2. **Explicit boundaries:** semantic workout work is contained in `::: block` directives.
3. **Preserve raw source:** a parser or converter must keep original text when normalization is uncertain.
4. **Unknown-safe:** unknown fields and kinds should be preserved, not discarded.
5. **AI-auditable:** an AI converter should produce output a human can review without hidden assumptions.

## Document Shape

A structured Workout Day Markdown document should start with one H1 title and contain one or more ordered workout blocks.

```md
# Hypertrophy System S7.3

::: block id=part-3 name="Lower Body" kind=tri-set
complete: 4 rounds
rest: 1m30s-2m
rpe: 8

- Eccentric Nordic Hamstring Curl
  reps: 5
  tempo: "Slow the eccentric as much as possible"

- Leg Extension
  reps: 10/side
  tempo: 20X2

- Banded Clamshell
  reps: 10/side
:::
```

### Required For A Structured Document

- One H1 title.
- At least one `::: block` directive.
- A `kind` attribute on every block.

### Plain Markdown

Plain Markdown before, between, or after blocks is allowed. It is document-level context unless it is inside a block.

Applications may render document-level context, but it is not part of the required completion list unless it appears inside a block.

## Directive Syntax

Workout blocks use Markdown container directives:

```md
::: block key=value key="quoted value"
body
:::
```

Rules:

- The directive name is `block`.
- Attribute order is not significant.
- Unquoted values may not contain whitespace.
- Quoted values may contain whitespace.
- A block closes with `:::`.
- Blocks are completed in document order.

## Block Model

A block is one ordered unit of work to complete during the workout day.

### Block Attributes

Required:

- `kind`: semantic completion style.

Recommended:

- `id`: stable local identifier within the day.
- `name`: display label.

Example:

```md
::: block id=bench-main name="Bench Press" kind=strength
sets: 5
rest: 3m
:::
```

### Block Fields

Fields are written as `key: value` lines inside a block before or between exercise items.

Common block fields:

- `complete`: human-readable completion rule.
- `sets`: number or range of sets.
- `rounds`: number or range of rounds.
- `duration`: total work duration.
- `cap`: time cap.
- `interval`: interval cadence.
- `work`: work duration inside intervals.
- `rest`: rest prescription.
- `rpe`: target RPE.
- `rir`: target reps in reserve.
- `tempo`: default tempo for the block.
- `score`: expected score type.
- `note`: short block note.
- `instructions`: longer coaching instruction.

Unknown fields are valid and should be preserved.

### Standard Block Kinds

Known `kind` values:

- `warmup`
- `mobility`
- `activation`
- `strength`
- `hypertrophy`
- `skill`
- `superset`
- `tri-set`
- `circuit`
- `complex`
- `amrap`
- `emom`
- `interval`
- `tabata`
- `for-time`
- `test`
- `finisher`
- `cooldown`
- `instruction`

Unknown kinds are valid custom kinds. Parsers should preserve them.

## Exercise Items

Exercises are Markdown list items inside a block. The list item text is the exercise display name. Markdown links may preserve source media.

```md
- [Cable Horizontal Narrow Grip Pull](https://youtu.be/example)
  reps: 10-12
  tempo: 20X2
```

Common exercise fields:

- `reps`: reps, rep range, max reps, side split, or alternation.
- `duration`: work duration or hold duration.
- `distance`: distance prescription.
- `calories`: calorie prescription.
- `load`: absolute load, range, bodyweight, or equipment load.
- `intensity`: percent, percent range, RPE-derived target, velocity, or training-max reference.
- `tempo`: tempo code or text.
- `rest`: rest after this item.
- `rpe`: item-specific RPE.
- `rir`: item-specific reps in reserve.
- `side`: left, right, each, alternating, or side-specific text.
- `equipment`: equipment prescription.
- `note`: short item note.
- `instructions`: longer item instruction.

Unknown exercise fields are valid and should be preserved.

### Composite Items

When a complex or mini-sequence is completed as one item inside a larger block, represent it as a normal exercise item with ordered `component N` fields. This keeps the parent block flat while preserving the sub-work in order.

```md
- Loaded Core Complex
  component 1: 10s Loaded Hollow Hold
  component 2: 10s Loaded Flutter Kick
  component 3: 10s Loaded Hollow Rock
```

Use this convention when the sub-parts are not independently scored or tracked. If the sub-parts need their own completion rules, split them into separate blocks instead.

## Prescription Values

The format intentionally accepts raw training strings. These values are valid:

```md
reps: 8
reps: 10-12
reps: 10/side
reps: 8/8
reps: 10+10
reps: 12 alternating
reps: max
duration: max hold
duration: 30-45s
distance: 20m
load: bodyweight
load: PDC
intensity: 68% 1RM
intensity: 65-70% 1RM
intensity: 80% + 2.5kg
tempo: 31X1
tempo: slow eccentric
rest: 1m30s-2m
```

Parsers may normalize values into typed data, but they must preserve the source string.

Equal slash-pair reps such as `8/8` are valid left/right per-side shorthand and may normalize to the shared count, equivalent to `8/side`. Unequal slash pairs such as `8/10` are ambiguous unless surrounding context defines them, so parsers should preserve the raw value instead of silently flattening it.

## Timed And Scored Blocks

### AMRAP

```md
::: block id=conditioning name="AMRAP 35" kind=amrap duration=35m
score: rounds+reps

- Strict Chest-to-Bar
  reps: 3

- Wall-Facing Handstand Hold
  duration: 30s

- Clapping Push-ups
  reps: 10
:::
```

### EMOM

```md
::: block id=emom-12 name="EMOM 12" kind=emom duration=12m interval=1m
score: completed rounds

minute 1:
- Row
  calories: 12

minute 2:
- Burpees
  reps: 10

minute 3:
- DB Snatch
  reps: 8/side
:::
```

Slot labels such as `minute 1:` group following exercise items until the next slot label or the end of the block.

### For Time

```md
::: block id=metcon name="For Time" kind=for-time cap=15m
score: time

- Back Squat
  reps: 21-15-9
  load: 60kg

- Pull-up
  reps: 21-15-9
:::
```

### Tabata Or Interval

```md
::: block id=tabata-bike name="Bike Tabata" kind=tabata rounds=8 work=20s rest=10s
score: calories

- Air Bike
:::
```

### Test

```md
::: block id=max-pull-up name="Pull-up Test" kind=test
score: reps

- Strict Pull-up
  reps: max
  rest: 3m

- Chin-up
  reps: max
  rest: 3m
:::
```

## Sequencing And Relationships

The workout day is completed in document order.

Relationships are expressed through block kind and fields:

- `superset`, `tri-set`, `circuit`, and `complex` imply repeated grouped work.
- `amrap`, `emom`, `interval`, `tabata`, and `for-time` imply time or score rules.
- `instruction` can describe setup, skill practice, or non-exercise work.

v0.1 does not define cross-block dependencies or conditional branching.

## AI Conversion Rules

When converting from PDFs, screenshots, spreadsheets, coach notes, or app-specific formats:

1. Output one Workout Day Markdown document per workout day.
2. Do not invent missing values.
3. Preserve ambiguous text in `note` or `instructions`.
4. Prefer known fields when the mapping is obvious.
5. Preserve unknown labels as custom fields.
6. Use known `kind` values when clear; otherwise preserve the source label as a custom kind.
7. Keep movement names close to the source text unless normalization is explicitly requested.
8. Preserve source media links when available.
9. If a prescription cannot be parsed safely, keep it as a raw field string.
10. The output should be reviewable by a human without the original source.

## Conformance

A conforming v0.1 producer should emit:

- a title;
- ordered `block` directives;
- block `kind` attributes;
- exercise list items for exercise work;
- raw strings for uncertain values.

A conforming v0.1 parser should:

- parse blocks in order;
- parse block attributes;
- parse block and exercise field lines;
- preserve unknown kinds and fields;
- preserve raw source text;
- tolerate plain Markdown outside blocks.
